Severity: Critical  
Category: Bugs  
Title: Unsupported `command_contains_any` causes four Windows rules to overmatch  
Detail: Four rules express OR logic through `command_contains_any`, but `_matches_rule()` never evaluates that field. Because the schema accepts unknown detection properties, these rules load successfully and then match primarily on `process_name`. This turns routine PowerShell, MSHTA, Regsvr32, and WMIC execution into alerts. The scorer also ignores the missing criterion, so matching and scoring disagree about what the rule represents.  
Recommendation: Implement case-insensitive OR matching in `DetectionEngine._matches_rule()`. Define the field as a non-empty string array in the schema. Count it in `_count_detection_criteria()`. Add positive, negative, empty-list, mixed-case, and combined-criteria tests for all four affected rules.  
File(s): `core/engine.py`, `core/scorer.py`, `rules/schema.json`, four affected files under `rules/windows/`, detection engine and rule validation tests

2.

Severity: High  
Category: Bugs  
Title: Unknown detection fields are silently accepted and then ignored  
Detail: The detection schema does not set `additionalProperties` to `false`. Any misspelled or unsupported operator therefore passes validation. `_matches_rule()` also returns `True` when it recognizes no effective criteria. Together, these behaviors fail open: a malformed rule can become substantially broader than its author intended. The confirmed bug is one instance of this larger design defect.  
Recommendation: After defining every supported operator, set `additionalProperties: false`. Require at least one detection property with `minProperties: 1`. Add a defensive runtime check that rejects unsupported or empty detection blocks rather than treating them as matches. Validate every bundled rule in CI.  
File(s): `rules/schema.json`, `core/engine.py`, `core/rule_loader.py`, tests

3.

Severity: High  
Category: Security  
Title: API has no visible authentication and permits unrestricted cross-origin access  
Detail: `CORS(app)` allows browser clients from arbitrary origins. No authentication or authorization is visible in the application setup. If scan, rule-management, ingestion, or alert endpoints are writable, any network client reaching port 5000 may be able to invoke them. Even read-only access can expose usernames, commands, paths, and incident details. CORS is not an authentication control, but unrestricted CORS makes browser-based abuse easier.  
Recommendation: Bind to loopback by default. Require authentication on all non-health endpoints, preferably a hashed API token initially, with constant-time comparison and clear read versus administrative scopes. Restrict CORS to configured dashboard origins, or disable it when the dashboard is same-origin. Add rate limits and tests proving unauthorized requests fail. Never store a plaintext token in tracked configuration.  
File(s): `api/server.py`, API route modules, `core/config.py`, deployment documentation, API tests

4.

Severity: High  
Category: Architecture  
Title: One SQLite connection is shared across Flask request threads  
Detail: `check_same_thread=False` only disables SQLite’s Python thread-affinity check. It does not make a shared connection, cursor usage, or multi-statement transactions safe as application-level concurrent operations. Requests can interleave transaction state and trigger lock errors or inconsistent commits. A mutex around writes helps, but a single global connection remains brittle. Claims of inevitable database corruption would be overstated, but the current design is unsafe for concurrent service use.  
Recommendation: Use one connection per request or thread, close it at context teardown, enable WAL mode, configure `busy_timeout`, use explicit short transactions, and serialize schema migration. Keep parameterized statements throughout. PostgreSQL should be documented as the production path if multi-user ingestion is intended.  
File(s): `core/database.py`, `api/server.py`, database concurrency tests

5.

Severity: High  
Category: Security  
Title: Log source validation allows arbitrary filesystem paths  
Detail: `validate_source()` accepts any existing readable file or directory. If an API request can control `source`, the service can be used to probe or parse files available to its operating-system account. This is an authorization flaw even when `../` is not present because absolute paths and symlinks can escape an intended log directory. Opening a file in text mode is also a weak readability test and may mishandle binary or special files.  
Recommendation: Resolve both the configured source root and candidate path with `Path.resolve(strict=True)`, then enforce containment with `candidate.is_relative_to(root)`. Reject symlink escapes, device files, FIFOs, sockets, unsupported extensions, oversized files, and directories unless a collector explicitly supports directories. Pass validated `Path` objects downstream.  
File(s): `collectors/base.py`, API ingestion routes, `core/config.py`, collector security tests

Severity: High  
Category: Bugs  
Title: Parent-process score bonus measures telemetry presence rather than evidence  
Detail: The scorer adds 15 points whenever an event contains `parent_process_name`, even if the matching rule does not constrain the parent. Hosts with richer telemetry therefore receive higher alert scores for identical behavior. Scores can vary because a field exists, not because suspicious evidence was matched.  
Recommendation: Remove this bonus or award it only when a rule contains a parent-process predicate that matched. More broadly, score matched evidence explicitly instead of inferring confidence from populated event fields. Add score invariance tests for irrelevant optional telemetry.

7.

Severity: High  
Category: Detection Quality  
Title: Static bonus stacking makes the 0 to 150 score poorly calibrated  
Detail: Severity, number of configured criteria, presence of a parent, and number of MITRE mappings are added together. Criteria count and ATT&CK tag count are rule metadata, not independent evidence of maliciousness. A high-severity rule with one criterion, parent telemetry, and several ATT&CK tags can immediately hit the 150 cap. Different raw totals then collapse to the same output, destroying ranking resolution. The score is effectively predetermined by rule authoring rather than event context.  
Recommendation: Separate rule severity from confidence. Use a documented 0 to 100 confidence score based on matched evidence, context, prevalence, allowlist status, and correlation. Treat ATT&CK IDs as labels, not points. If retaining 0 to 150 for compatibility, define operational bands and expose a score breakdown. Calibrate against a labeled benign and malicious corpus.  
File(s): `core/scorer.py`, rule files, API response schema, dashboard, documentation, scorer tests

8.

Severity: Medium  
Category: Detection Quality  
Title: Process and parent matching use unsafe substring semantics  
Detail: `expected_process in actual_process` can match unintended names such as `notpowershell.exe`, while parent and path allowlists use similar substring checks. Path checks against `working_directory` do not establish that the executable originated from an approved path. This produces both false positives and broad bypass opportunities.  
Recommendation: Normalize process names and compare basenames exactly, with platform-appropriate case handling. Treat executable path, working directory, and process name as separate event fields. For path allowlists, normalize separators and resolve canonical paths where safe, then use path-component containment instead of raw substring matching.  
File(s): `core/engine.py`, `collectors/base.py`, platform collectors, rules, matching tests

9.

Severity: Medium  
Category: Security  
Title: Rule-controlled regular expressions can cause denial of service  
Detail: `command_regex` and `user_pattern` are evaluated directly with Python `re` against attacker-influenced log fields. A rule containing catastrophic backtracking can consume substantial CPU. Invalid expressions also fail at event-processing time unless caught elsewhere. This becomes more relevant if rules are downloaded or editable through the API.  
Recommendation: Compile all regular expressions during rule loading and reject invalid patterns before activation. Impose pattern and input-length limits. Prefer a timeout-capable regex implementation or restrict the supported regex subset. Cache compiled predicates in the `Rule` object.  
File(s): `core/rule_loader.py`, `core/engine.py`, `rules/schema.json`, rule validation tests
