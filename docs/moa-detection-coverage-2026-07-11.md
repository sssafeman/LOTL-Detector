Important engine constraint: each YAML mapping can contain only one `command_contains_any` key. Repeating that key silently overwrites earlier groups in standard YAML parsers. The rules below avoid duplicate keys and stay within the documented operators.

## 1. Coverage gap analysis

### Windows gaps

| Gap | ATT&CK | Why it matters |
|---|---|---|
| PowerShell download cradles | T1059.001, T1105 | Encoded command detection does not catch cleartext in-memory download and execution. |
| Rundll32 script execution | T1218.011 | A signed Microsoft binary can proxy JavaScript execution and bypass application controls. |
| SAM hive extraction | T1003.002 | Stealing SAM and SYSTEM hives enables offline local credential recovery. |
| Remote MSI execution | T1218.007 | `msiexec.exe` can retrieve and execute attacker-controlled packages. |
| CMSTP proxy execution | T1218.003 | Malicious INF files can execute commands through a trusted Windows utility. |

Other later candidates include MSBuild inline tasks, InstallUtil, Office child-process chains, PowerShell downgrade attempts, and `rundll32.exe` MiniDump credential access.

### Linux gaps

| Gap | ATT&CK | Why it matters |
|---|---|---|
| Python reverse shells | T1059.006 | Python is commonly available and frequently used after web application compromise. |
| Systemd persistence | T1543.002 | Malicious service units provide privileged, reboot-persistent execution. |
| LD_PRELOAD hijacking | T1574.006 | Attackers can inject shared objects into otherwise legitimate processes. |
| SSH key persistence | T1098.004 | Injected authorized keys provide quiet, passwordless persistence. |
| Wget download and execution | T1105, T1059.004 | Curl coverage does not cover the equally common wget staging chain. |

Other later candidates include Python HTTP servers, Perl reverse shells, `at` persistence, shell profile modification, namespace abuse, `find` or `awk` shell escape, and sensitive archive staging.

### `WIN-007.yaml`

```yaml
name: "PowerShell WebClient Download Cradle"
id: "WIN-007"
platform: windows
severity: high
mitre_attack:
  - T1059.001
  - T1105
description: |
  Detects PowerShell downloading content through Net.WebClient and passing
  the result to Invoke-Expression. This is a common in-memory payload staging
  pattern used during initial access, lateral movement, and command execution.
detection:
  process_name: "powershell.exe"
  command_contains:
    - "Net.WebClient"
    - "DownloadString"
  command_contains_any:
    - "IEX"
    - "Invoke-Expression"
false_positives:
  - "Legacy administrative scripts that load trusted modules from an internal web server."
  - "Authorized adversary simulation and penetration testing."
whitelist:
  parent_processes:
    - "DeployAgent.exe"
  users:
    - "svc_automation"
  paths:
    - "C:\\ProgramData\\Company\\Deployment\\"
response:
  - "Capture the complete PowerShell command and extract the remote URL."
  - "Review proxy, DNS, and endpoint network events for the destination."
  - "Determine whether the downloaded content executed in memory or wrote additional files."
  - "Trace the parent and grandparent processes to identify the initial execution vector."
  - "Isolate the endpoint if the URL or execution chain is not authorized."
```

Justification: encoded PowerShell is only one evasion form. Cleartext WebClient cradles remain common and are materially different from WIN-002.

### `LNX-007.yaml`

```yaml
name: "Python Reverse Shell One-Liner"
id: "LNX-007"
platform: linux
severity: critical
mitre_attack:
  - T1059.006
description: |
  Detects a Python command line combining socket creation, file descriptor
  duplication, and subprocess execution. This combination is characteristic
  of Python reverse shells used after web application or service compromise.
detection:
  command_contains:
    - "socket"
    - "dup2"
    - "subprocess"
  command_contains_any:
    - "python -c"
    - "python3 -c"
    - "python2 -c"
  command_regex: '(?i)python[0-9.]*\s+-c\s+.*socket.*dup2.*subprocess'
false_positives:
  - "Authorized penetration testing or exploit validation."
  - "A rare diagnostic one-liner that intentionally redirects process descriptors over a socket."
whitelist:
  parent_processes:
    - "security-test-runner"
  users:
    - "redteam"
  paths:
    - "/opt/company/security-tests/"
response:
  - "Immediately identify and block the remote IP address and port."
  - "Capture the complete process tree and active network connections."
  - "Determine which exposed service or user session launched Python."
  - "Preserve relevant shell history, web logs, temporary files, and process telemetry."
  - "Isolate the host when the activity is not an approved security test."
```

Justification: the three-component combination is highly specific to shell redirection and avoids noisy alerts on ordinary Python networking.

Note: `process_name` is intentionally omitted because the documented operator cannot express `python`, `python2`, and `python3` as alternatives.

## 3. Priority ranking

### 1. WIN-009: Registry Hive Export

Why first:

* Critical credential-access behavior.
* Very low false-positive rate.
* Simple matching with little parser ambiguity.
* Detects a point where an intrusion is likely transitioning toward privilege expansion or lateral movement.

### 2. LNX-007: Python Reverse Shell

Why second:

* Strong combination of `socket`, `dup2`, and `subprocess`.
* Common after exploitation of web applications and exposed services.
* Immediate containment value because it exposes an interactive attacker channel.
* Extremely low expected noise outside security testing.

### 3. WIN-007: PowerShell WebClient Download Cradle

Why third:

* Fills a direct gap in the existing encoded-command rule.
* Covers payload retrieval plus in-memory execution.
* Frequently appears in initial access, lateral movement, and attacker tooling.
* Stronger signal than broad PowerShell flag detection.

WIN-008 is a close fourth because its specific MSHTML pattern is also nearly noise-free.

## 4. False-positive risk and tuning guidance

| Rule | Risk | Tuning guidance |
|---|---|---|
| WIN-007 | Low | Baseline deployment systems. Whitelist a verified parent, account, and internal URL workflow together rather than excluding all automation users. |
| WIN-008 | Very low | No general tuning should be necessary. Suppress only approved security-test harnesses. |
| WIN-009 | Very low | Validate backup and forensic tools. Do not whitelist all SYSTEM activity because attackers commonly execute as SYSTEM. |
| WIN-010 | Low to medium | Inventory approved remote MSI repositories and deployment parents. Prefer domain-aware tuning when URL fields become available. |
| WIN-011 | Low | Inventory CMSTP-dependent VPN software. Whitelist a precise installer and INF directory, not all CMSTP use. |
| LNX-007 | Very low | Suppress only controlled penetration-test accounts and runners. A broad developer whitelist would hide real compromises. |
| LNX-008 | Medium | Package managers and configuration management are the main noise sources. Correlate with package transactions and signed deployment jobs. |
| LNX-009 | Low | Tune known profilers and build systems. Temporary shared objects used outside those workflows should remain alertable. |
| LNX-010 | Medium | Key management produces legitimate matches. Compare inserted fingerprints with an approved inventory before suppression. |
| LNX-011 | Low to medium | CI bootstrap commands can match. Restrict exclusions to trusted runners, repositories, users, and working directories. |

Whitelist caution: parent process ancestry differs across telemetry sources. A package manager may be a grandparent rather than the immediate parent. Verify actual event data before relying on these sample exclusions.

## Implementation notes

1. Make command matching case-insensitive on Windows. Linux command matching should remain case-sensitive unless a rule regex explicitly says otherwise.
2. Normalize Windows process names to their basename before comparison.
3. Compile regexes during rule loading. Reject invalid regexes immediately rather than failing during event processing.
4. Define whitelist semantics explicitly. Recommended behavior is suppression when any populated whitelist category matches, but require exact normalized values rather than loose substrings.
5. Record both `matched_rule` and `whitelist_reason` in debug mode. Otherwise tuning becomes archaeology.
6. Add negative tests where only part of an AND condition is present.
7. Add case-variation tests for Windows commands.
8. Do not broadly whitelist `SYSTEM`, `root`, PowerShell, Bash, package managers, or temporary paths. Those are precisely the contexts attackers seek.
