Recommendation: use a multiplicative impact and confidence model. An additive model still allows rule severity to dominate when event evidence is weak. The score should represent risk, not severity with miscellaneous bonuses attached.

One correction to the supplied reference calculations: with the proposed formula, critical severity at 50 confidence scores 94, not 113. High severity reaches the high band at roughly 74 confidence, not 87. The specification below uses the corrected mathematics.

1. Scoring algorithm

Maintain three values:

• severity_subscore: 25, 50, 75, or 100  
• confidence_subscore: integer from 0 to 100  
• score: final integer from 0 to 150  

Exact formula:

```text
risk_multiplier = 0.25 + (0.75 × confidence_subscore / 100)

raw_score = severity_subscore × risk_multiplier × 1.5

score = clamp(round_half_up(raw_score), 0, 150)
```

Use round half up explicitly. Do not rely on Python `round()`, because Python uses banker's rounding.

Equivalent integer friendly formula:

```text
raw_numerator = severity_subscore × (25 + 0.75 × confidence_subscore)
```

The decimal implementation is clearer and less error prone:

```text
score = floor(raw_score + 0.5)
```

Behavior:

| Severity | Confidence 0 | Confidence 25 | Confidence 50 | Confidence 75 | Confidence 100 |
|---|---:|---:|---:|---:|---:|
| Critical, 100 | 38 | 66 | 94 | 122 | 150 |
| High, 75 | 28 | 49 | 70 | 91 | 113 |
| Medium, 50 | 19 | 33 | 47 | 61 | 75 |
| Low, 25 | 9 | 16 | 23 | 30 | 38 |

Why retain the 0.25 floor:

• Severity still matters when evidence is weak.  
• A critical rule with zero confidence remains visible at 38.  
• Weak evidence cannot produce a high priority alert.  
• Low impact activity cannot become critical merely through many confidence bonuses.

Clamp behavior:

```text
severity_subscore = clamp(severity_subscore, 0, 100)
confidence_subscore = clamp(raw_confidence, 0, 100)
score = clamp(round_half_up(raw_score), 0, 150)
```

Clamping is applied after all confidence factors. Individual factor groups are capped before combination.

2. Confidence subscore

Confidence begins at 35, representing the minimum reasonable confidence for an alert that passed rule matching.

```text
raw_confidence =
    35
    + criteria_strength
    + parent_context
    + user_context
    + command_anomaly
    + whitelist_adjacency
    + admin_context
    + partial_match_penalty

confidence_subscore = clamp(raw_confidence, 0, 100)
```

A score of 50 is neutral. Scores above 70 indicate strong event evidence. Scores below 30 indicate weak or benign adjacent evidence.

2.1 Detection criteria strength

Range: 0 to positive 30.

Only criteria actually evaluated as true contribute. Merely being configured on a rule contributes nothing.

Use the current four criterion types:

| Matched criteria count | Delta |
|---:|---:|
| 0 | 0 |
| 1 | 5 |
| 2 | 16 |
| 3 | 25 |
| 4 | 30 |

This is intentionally nonlinear. The second independent criterion provides more corroboration than the fourth.

Important implementation rule:

The detector must pass the scorer the criteria evaluation result. The scorer must not reimplement rule matching. It should receive:

```text
configured_criteria
matched_criteria
partially_matched_criteria
```

Example:

```text
configured_criteria = [
    "command_contains",
    "command_regex",
    "parent_process"
]

matched_criteria = [
    "command_contains",
    "parent_process"
]
```

Duplicate entries do not count. A rule cannot inflate confidence by declaring multiple equivalent patterns under the same criterion type.

If multiple regexes exist inside `command_regex`, the entire criterion type still counts once.

2.2 Parent and child process consistency

Range: negative 5 to positive 15. Exactly one condition applies.

| Condition | Delta |
|---|---:|
| Parent telemetry unavailable | 0 |
| Parent matches the rule's suspicious parent criterion | positive 15 |
| Parent does not match the rule, but lineage is independently classified as anomalous | positive 8 |
| Parent is present but neither expected nor anomalous | 0 |
| Parent and child combination matches a configured common benign lineage | negative 5 |

Examples of configurable benign lineages:

```text
explorer.exe to powershell.exe
services.exe to svchost.exe
services.exe to configured service binary
taskeng.exe to approved scheduled task binary
```

Do not award points merely because a parent exists. That was the original design flaw.

Do not globally classify `explorer.exe`, `cmd.exe`, or `services.exe` as benign without considering the child. Parent legitimacy is a relationship, not a filename lookup.

Evaluation precedence:

1. Suspicious rule parent match  
2. Independently anomalous lineage  
3. Configured benign lineage  
4. Neutral  

This ensures a rule specifically looking for `winword.exe` spawning PowerShell receives positive 15 even if office applications are common in the environment.

2.3 User context

Range: 0 to positive 15. Only the strongest applicable condition applies.

| Condition | Delta |
|---|---:|
| Service or SYSTEM account launches an interactive shell outside an approved service lineage | positive 15 |
| Non administrator executes a tool or action requiring elevated privilege | positive 12 |
| User differs from the established owner of the process, task, or service | positive 8 |
| User is unusual for this host or process according to configured baseline | positive 5 |
| No anomaly or context unavailable | 0 |

Do not treat SYSTEM as inherently suspicious. SYSTEM running `services.exe` is normal. SYSTEM running an interactive encoded PowerShell command under an unusual parent is suspicious.

Context unavailable must produce zero, not a penalty.

2.4 Command line anomaly indicators

Range: 0 to positive 25.

Weights are additive, then capped at 25. Each category counts once even if several patterns inside it match.

| Indicator category | Delta |
|---|---:|
| Encoded payload or command | positive 12 |
| Download or execution cradle | positive 10 |
| Obfuscation or escaping | positive 8 |
| Unusual executable or payload path | positive 5 |
| Suspicious hidden or bypass flags | positive 5 |
| Long or high entropy argument | positive 4 |

Exact categories:

Encoded payload or command:

```text
PowerShell EncodedCommand or enc
FromBase64String
certutil decode
base64 decode followed by execution
```

Download or execution cradle:

```text
Invoke Expression
DownloadString
DownloadFile
Net.WebClient
Start BitsTransfer followed by execution
curl or wget output followed by execution
```

Obfuscation or escaping:

```text
PowerShell backtick escaping
caret escaping in cmd
environment variable reconstruction
string concatenation used to construct command names
character code reconstruction
```

Unusual path:

```text
temporary directories
user writable AppData paths
public user directory
recycle bin
UNC or WebDAV execution path
```

Suspicious flags:

```text
hidden window
execution policy bypass
noninteractive plus hidden
rundll32 script protocol use
regsvr32 remote scriptlet use
```

Long or high entropy:

```text
argument length at least 300 characters
or entropy at least 4.5 bits per character over a token of at least 80 characters
```

Examples:

• Encoded command alone: positive 12  
• Encoded command plus cradle: positive 22  
• Encoded command, cradle, and obfuscation: raw 30, capped at positive 25  

These patterns should be detected by a dedicated command analyzer. Avoid embedding a large pattern library directly in `Scorer`.

2.5 Whitelist adjacency

Range: negative 30 to 0. Only the strongest applicable condition applies.

| Condition | Delta |
|---|---:|
| Exact whitelist match reaching the scorer | negative 30 |
| High similarity to approved command | negative 20 |
| Moderate similarity to approved command | negative 10 |
| No similarity | 0 |

Exact whitelist matches should normally be suppressed before alert creation. The negative 30 value is a defensive fallback.

Similarity definitions:

High similarity:

```text
normalized token Jaccard similarity at least 0.90
and executable, parent, and user scopes match
```

Moderate similarity:

```text
normalized token Jaccard similarity at least 0.70
and executable plus at least one scope field match
```

Do not use raw character edit distance as the primary comparison. Paths, usernames, timestamps, and identifiers make it unstable. Normalize first:

• lowercase executable and flags  
• replace quoted paths with path tokens where appropriate  
• replace GUIDs, timestamps, hashes, and numeric identifiers with typed placeholders  
• preserve security relevant flags  
• tokenize using the platform's command line parser  

Whitelist scope must support:

```text
executable
parent_process
user or user_group
host or host_group
command_pattern
valid_from
valid_until
```

A similar command from the wrong user or host must not receive the full penalty.

2.6 Common administrative context

Range: negative 20 to 0. Exactly one condition applies.

| Condition | Delta |
|---|---:|
| Known admin user, approved admin host, business hours, approved tool | negative 20 |
| Three of those four conditions | negative 10 |
| Known admin user with only one additional condition | negative 5 |
| Context unavailable or not administrative | 0 |

The four conditions are:

1. User belongs to configured administrative identity set  
2. Source host belongs to configured administrative workstation set  
3. Event occurs inside configured maintenance or business window  
4. Executable or command belongs to approved administrative tooling  

Do not subtract points merely because the username contains `admin` or the process is PowerShell.

Avoid double counting. User anomaly and admin suppression represent opposite interpretations of identity context. Apply this rule:

```text
if user_context > 0:
    admin_context = 0
```

A context cannot simultaneously be classified as anomalous and as fully approved administrative activity.

2.7 Partial match penalty

Range: negative 15 to 0.

This is separate from matched criteria count because a weak partial value can cause a permissive detector to fire.

| Condition | Delta |
|---|---:|
| Rule uses AND semantics and one or more required criteria did not match | negative 15 |
| Rule uses OR semantics, only one of three or more criteria matched | negative 8 |
| One criterion matched partially rather than exactly | negative 5 |
| Full rule match | 0 |

Only the strongest penalty applies.

A partial match must be explicitly reported by the detector. The scorer must not infer partial matching by rerunning regexes.


3. Severity subscore

Severity is static and rule defined:

| Rule severity | Subscore |
|---|---:|
| low | 25 |
| medium | 50 |
| high | 75 |
| critical | 100 |

Unknown severity handling:

Do not silently default unknown values to low. That hides rule authoring defects.

Recommended behavior:

```text
strict mode:
    raise InvalidRuleSeverity

compatibility mode:
    use 25
    include validation warning in breakdown
```

Production rule loading should run in strict mode. Compatibility mode is appropriate only for reading legacy data.

MITRE ATT&CK treatment:

MITRE technique count contributes exactly zero.

MITRE tags remain in:

• rule metadata  
• alert metadata  
• dashboard filtering  
• ATT&CK coverage reports  
• correlation logic  

MITRE count measures classification breadth, not impact or likelihood. A rule tagged with two overlapping techniques is not twice as dangerous.

If the project later needs technique specific impact, introduce an explicit rule field such as `impact_profile`. Do not derive impact from tag count.

Requirements for the breakdown:

• Every factor group must appear, including zero contributions.  
• `delta` must always be a signed integer.  
• Use machine stable reason codes, not prose as the primary identifier.  
• Evidence must contain normalized values, not secret or sensitive enrichment data.  
• Store the scoring version. Otherwise historical scores cannot be audited after weight changes.  
• Preserve raw and clamped confidence. Clamping can otherwise conceal how strongly factors saturated.  
• Do not store the formula as arbitrary executable text. Store a stable formula version identifier.

For list endpoints, allow:

```text
include_score_breakdown=false
```

The score, risk band, severity subscore, confidence subscore, and scoring version should remain present.

5. Threshold reinterpretation

The dashboard label should be called `risk_band`, not `severity`. Rule severity and final risk are different concepts.

Recommended thresholds:

| Final score | Risk band | Interpretation |
|---:|---|---|
| 120 to 150 | critical | Critical impact with strong evidence |
| 90 to 119 | high | High investigation priority |
| 60 to 89 | medium | Meaningful impact and evidence, analyst review warranted |
| 0 to 59 | low | Low impact, weak evidence, or benign adjacent context |

Exact confidence requirements, subject to integer rounding:

| Rule severity | Medium, score at least 60 | High, score at least 90 | Critical, score at least 120 |
|---|---:|---:|---:|
| Critical | confidence about 20 | confidence about 47 | confidence about 74 |
| High | confidence about 38 | confidence about 74 | impossible |
| Medium | confidence about 74 | impossible | impossible |
| Low | impossible | impossible | impossible |

This gives the bands useful invariants:

• Only critical rules can produce critical risk.  
• High rules require strong confidence to produce high risk.  
• Medium rules can reach medium risk only with strong confidence.  
• Low rules remain low regardless of confidence.  

That last property is intentional. If a low severity rule with perfect evidence should become medium priority, its rule impact is incorrectly classified.

6. Migration plan

6.1 `core/scorer.py`

Remove scoring contributions from:

```text
DETECTION_CRITERION_BONUS
PARENT_PROCESS_BONUS
MITRE_TECHNIQUE_BONUS
```

Add:

```text
SEVERITY_SUBSCORES
CRITERIA_DELTAS
COMMAND_ANOMALY_WEIGHTS
CONFIDENCE_BASE
CONFIDENCE_MIN
CONFIDENCE_MAX
RISK_BAND_THRESHOLDS
SCORING_VERSION
```

Introduce structured inputs:

```text
MatchEvidence
ParentContext
UserContext
CommandAnalysis
WhitelistSimilarity
AdminContext
```

The scorer should consume already evaluated facts. It should not:

• rerun rule regexes  
• query a directory service  
• determine business hours  
• perform whitelist similarity searches  
• infer whether the user is an administrator  
• maintain environment baselines  

Those belong in the detector or context enrichment layer. Keeping them outside `Scorer` makes scoring deterministic and easy to test.

Public API:

```text
score(rule, evidence) -> ScoreResult
```

For a short transition, retain:

```text
score_value(rule, evidence) -> int
```

This wrapper returns `ScoreResult.score`.

Do not keep two independent implementations. The compatibility method must delegate to the new method.

6.2 Alert dataclass

Keep:

```text
score: int
severity: str

```

Add:

```text
risk_band: str
severity_subscore: int

```

Recommended field defaults for reading legacy alerts:

```text

```

6.3 Database

The requirement only mandates compatibility with the existing integer score. Prefer the smallest schema change.

Option A, recommended:

```text
score                 existing integer
scoring_version       new integer, default 1
severity_subscore     new nullable integer
confidence_subscore   new nullable integer
score_breakdown       new nullable JSON or text JSON
```

`risk_band` can be computed from `score` rather than stored. This prevents stale duplicated values when thresholds change.

If the database does not support JSON, store canonical JSON text. Validate it during serialization and insertion.

Existing rows:

```text
score remains unchanged
scoring_version = 1
severity_subscore = null
confidence_subscore = null
score_breakdown = null
```

Do not reverse engineer legacy confidence from:

```text
old score minus severity
```

That result would include MITRE count and telemetry bonuses and would therefore be mislabeled.

6.5 Detector pipeline

The detector should return both the decision and evidence:

```text
match_result.matched
match_result.configured_criteria
match_result.matched_criteria
match_result.partially_matched_criteria
match_result.match_semantics
```

Add a context enrichment stage that produces:

```text
parent classification
user anomaly classification
command anomaly indicators
whitelist similarity classification
approved admin context classification
```

All classifications should use stable reason codes.

6.6 Rollout

Phase 1: implement version 2 behind a feature flag.

Phase 2: shadow score every new alert with versions 1 and 2 for at least one representative operational period. Store or log both scores, but continue presenting version 1.

Phase 3: compare:

• alert count per risk band  
• score distribution per rule  
• false positive rate per band  
• percentage of scores clamped at confidence 0 or 100  
• most frequent positive and negative factors  
• rules where version 1 and version 2 differ by at least 30 points  

Phase 4: tune only after reviewing labeled events. Do not tune weights merely to reproduce the old distribution. The old distribution is what is broken.

Phase 5: switch new alerts to version 2. Historical scores remain version 1.

Phase 6: remove the feature flag only after rollback is no longer needed. Keep version 1 deserialization permanently.

7. Concrete Python pseudocode

```python
from dataclasses import dataclass
from math import floor
from typing import Any


SEVERITY_SUBSCORES = {
    "low": 25,
    "medium": 50,
    "high": 75,
    "critical": 100,
}

CRITERIA_DELTAS = {
    0: 0,
    1: 5,
    2: 16,
    3: 25,
    4: 30,
}

COMMAND_ANOMALY_WEIGHTS = {
    "encoded_payload": 12,
    "download_cradle": 10,
    "obfuscation": 8,
    "unusual_path": 5,
    "suspicious_flags": 5,
    "high_entropy_argument": 4,
}

CONFIDENCE_BASE = 35
COMMAND_ANOMALY_CAP = 25
SCORING_VERSION = 2


@dataclass(frozen=True)
class MatchEvidence:
    configured_criteria: tuple[str, ...]
    matched_criteria: tuple[str, ...]
    partially_matched_criteria: tuple[str, ...]
    semantics: str
    required_criteria_missing: bool


@dataclass(frozen=True)
class ContextEvidence:
    parent_reason: str
    user_reason: str
    command_anomalies: tuple[str, ...]
    whitelist_reason: str
    admin_reason: str


@dataclass(frozen=True)
class ScoreResult:
    score: int
    risk_band: str
    severity_subscore: int
    confidence_subscore: int
    scoring_version: int
    breakdown: dict[str, Any]


class InvalidRuleSeverity(ValueError):
    pass


class Scorer:
    PARENT_DELTAS = {
        "missing_telemetry": 0,
        "suspicious_parent_match": 15,
        "anomalous_lineage": 8,
        "neutral_lineage": 0,
        "common_benign_lineage": -5,
    }

    USER_DELTAS = {
        "system_interactive_shell": 15,
        "non_admin_privileged_tool": 12,
        "unexpected_process_owner": 8,
        "unusual_user_baseline": 5,
        "neutral": 0,
        "missing_telemetry": 0,
    }

    WHITELIST_DELTAS = {
        "exact_match": -30,
        "high_similarity": -20,
        "moderate_similarity": -10,
        "no_match": 0,
        "not_evaluated": 0,
    }

    ADMIN_DELTAS = {
        "full_approved_context": -20,
        "three_conditions": -10,
        "two_conditions": -5,
        "not_admin_context": 0,
        "not_evaluated": 0,
    }

    def score(self, rule, match: MatchEvidence, context: ContextEvidence) -> ScoreResult:
        severity = self._severity(rule.severity)
        factors = []

        criteria_delta = self._criteria_delta(match)
        factors.append(self._factor(
            name="detection_criteria",
            delta=criteria_delta,
            reason=f"{len(set(match.matched_criteria))}_criteria_matched",
            evidence={
                "configured": sorted(set(match.configured_criteria)),
                "matched": sorted(set(match.matched_criteria)),
            },
        ))

        parent_delta = self.PARENT_DELTAS[context.parent_reason]
        factors.append(self._factor(
            "parent_context",
            parent_delta,
            context.parent_reason,
        ))

        user_delta = self.USER_DELTAS[context.user_reason]
        factors.append(self._factor(
            "user_context",
            user_delta,
            context.user_reason,
        ))

        anomaly_names = sorted(set(context.command_anomalies))
        anomaly_raw = sum(
            COMMAND_ANOMALY_WEIGHTS.get(name, 0)
            for name in anomaly_names
        )
        anomaly_delta = min(anomaly_raw, COMMAND_ANOMALY_CAP)
        factors.append(self._factor(
            name="command_anomaly",
            delta=anomaly_delta,
            reason=(
                "command_anomalies_detected"
                if anomaly_names
                else "none_detected"
            ),
            evidence={
                "indicators": anomaly_names,
                "uncapped_delta": anomaly_raw,
                "cap": COMMAND_ANOMALY_CAP,
            },
        ))

        whitelist_delta = self.WHITELIST_DELTAS[context.whitelist_reason]
        factors.append(self._factor(
            "whitelist_adjacency",
            whitelist_delta,
            context.whitelist_reason,
        ))

        if user_delta > 0:
            admin_delta = 0
            admin_reason = "not_applied_due_to_user_anomaly"
        else:
            admin_delta = self.ADMIN_DELTAS[context.admin_reason]
            admin_reason = context.admin_reason

        factors.append(self._factor(
            "admin_context",
            admin_delta,
            admin_reason,
        ))

        partial_delta, partial_reason = self._partial_match_delta(match)
        factors.append(self._factor(
            "partial_match",
            partial_delta,
            partial_reason,
            {
                "partially_matched": sorted(
                    set(match.partially_matched_criteria)
                )
            },
        ))

        raw_confidence = CONFIDENCE_BASE + sum(
            factor["delta"] for factor in factors
        )
        confidence = self._clamp(raw_confidence, 0, 100)

        confidence_multiplier = 0.25 + 0.75 * confidence / 100.0
        raw_score = severity * confidence_multiplier * 1.5
        rounded_score = floor(raw_score + 0.5)
        final_score = self._clamp(rounded_score, 0, 150)

        breakdown = {
            "severity": {
                "rule_severity": rule.severity,
                "subscore": severity,
                "mitre_techniques": list(rule.mitre_attack),
                "mitre_contribution": 0,
            },
            "confidence": {
                "base": CONFIDENCE_BASE,
                "raw": raw_confidence,
                "clamped": confidence,
                "factors": factors,
            },
            "combination": {
                "formula_version": "severity_confidence_v2",
                "severity": severity,
                "confidence": confidence,
                "confidence_multiplier": confidence_multiplier,
                "scale": 1.5,
                "raw_score": raw_score,
                "rounded_score": final_score,
            },
        }

        return ScoreResult(
            score=final_score,
            risk_band=self._risk_band(final_score),
            severity_subscore=severity,
            confidence_subscore=confidence,
            scoring_version=SCORING_VERSION,
            breakdown=breakdown,
        )

    def score_value(
        self,
        rule,
        match: MatchEvidence,
        context: ContextEvidence,
    ) -> int:
        return self.score(rule, match, context).score

    def _severity(self, severity_name: str) -> int:
        normalized = severity_name.lower()
        if normalized not in SEVERITY_SUBSCORES:
            raise InvalidRuleSeverity(severity_name)
        return SEVERITY_SUBSCORES[normalized]

    def _criteria_delta(self, match: MatchEvidence) -> int:
        matched_count = min(len(set(match.matched_criteria)), 4)
        return CRITERIA_DELTAS[matched_count]

    def _partial_match_delta(
        self,
        match: MatchEvidence,
    ) -> tuple[int, str]:
        if match.semantics == "and" and match.required_criteria_missing:
            return -15, "required_criteria_missing"

        configured_count = len(set(match.configured_criteria))
        matched_count = len(set(match.matched_criteria))

        if (
            match.semantics == "or"
            and configured_count >= 3
            and matched_count == 1
        ):
            return -8, "single_branch_of_broad_or"

        if match.partially_matched_criteria:
            return -5, "criterion_partially_matched"

        return 0, "full_rule_match"

    @staticmethod
    def _factor(
        name: str,
        delta: int,
        reason: str,
        evidence: dict | None = None,
    ) -> dict:
        return {
            "name": name,
            "delta": delta,
            "reason": reason,
            "evidence": evidence or {},
        }

    @staticmethod
    def _risk_band(score: int) -> str:
        if score >= 120:
            return "critical"
        if score >= 90:
            return "high"
        if score >= 60:
            return "medium"
        return "low"

    @staticmethod
    def _clamp(value: int, minimum: int, maximum: int) -> int:
        return max(minimum, min(maximum, value))
```

Final engineering recommendation:

Do not put environment analysis inside `Scorer`. Make the scorer a pure deterministic function over normalized evidence. That gives you reproducible alerts, stable unit tests, auditable breakdowns, and future weight calibration without entangling detection logic. The scorer should calculate risk. The detector and enrichment pipeline should establish facts.
