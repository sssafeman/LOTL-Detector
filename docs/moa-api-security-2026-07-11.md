# LOTL-Detector API Security Layer

## Executive decision

Use a static bearer API key, no RBAC, no authentication on a minimal health endpoint, strict CORS, localhost-only binding, per-request SQLite connections, WAL mode, schema validation, and in-memory rate limiting.

This fits the threat model:

- One local operator
- No user accounts
- No identity provider
- Sensitive detection data
- Resource-intensive scan endpoint
- Portfolio scope that must remain explainable

The current implementation has four material problems:

1. `CORS(app)` trusts every origin.
2. Every endpoint is unauthenticated.
3. The development server exposes itself on every interface with debug enabled.
4. A single SQLite connection is shared across request threads.

The global component instances are also unsafe for multiple application instances and make tests interfere with each other.

## 1. Authentication specification

### Scheme

Use:

```text
Authorization: Bearer *** key>
```

Do not use Basic authentication.

Reasons:

- There is no username or user model.
- Basic authentication implies account credentials and encourages browser credential caching.
- A bearer key maps directly to programmatic clients and static dashboard requests.
- OAuth2, sessions, JWTs, and refresh tokens add machinery without solving a real requirement.

A JWT is specifically the wrong choice. LOTL-Detector would still need to store a signing secret, and a self-issued JWT provides no meaningful advantage over an opaque random key.

### Key format

Generate 32 cryptographically random bytes, encoded using URL-safe Base64:

```python
secrets.token_urlsafe(32)
```

This produces approximately 256 bits of entropy.

Do not impose an exact encoded length during authentication. Validate only that the configured key is nonempty and has at least 32 characters.

### Key storage

Resolution order:

1. `LOTL_API_KEY` environment variable
2. File named by `LOTL_API_KEY_FILE`
3. Default file: `~/.config/lotl-detector/api.key`

The key file must:

- Be created with mode `0600`
- Be contained in a directory with mode `0700`
- Contain only the key and one trailing newline
- Never be committed to the repository
- Never be written to `config.yml`
- Never be logged

For test applications, inject `API_KEY` directly through the `create_app()` test configuration. Do not silently disable authentication whenever `TESTING=True`, because that makes the security behavior difficult to test.

Example test configuration:

```python
{
    "TESTING": True,
    "API_KEY": "test-k...ters",
}
```

### Startup behavior

Fail closed if authentication is enabled but no key can be loaded.

For first-run usability, a dedicated CLI command should generate the key:

```text
python -m api.auth generate
```

The server should not unexpectedly generate credentials during an import or test run. Explicit generation is easier to understand and avoids credentials being written to an unexpected home directory.

### Comparison

Use `hmac.compare_digest()`:

```python
hmac.compare_digest(provided.encode(), expected.encode())
```

Reject:

- Missing `Authorization` header
- Multiple `Authorization` headers
- Schemes other than `Bearer`
- Empty tokens
- Tokens containing whitespace
- Invalid tokens

Return the same response for missing and incorrect keys:

```json
{"error": "unauthorized"}
```

Status:

```text
401 Unauthorized
```

Include:

```text
WWW-Authenticate: Bearer
```

Do not reveal whether the key was missing, malformed, or incorrect.

### Protected endpoints

All endpoints require authentication except the reduced health endpoint.

| Endpoint | Method | Authentication |
|---|---:|---:|
| `/api/health` | GET | No |
| `/api/alerts` | GET | Yes |
| `/api/alerts/<id>` | GET | Yes |
| `/api/stats` | GET | Yes |
| `/api/rules` | GET | Yes |
| `/api/scan` | POST | Yes |
| Any future `/api/ingest` | POST | Yes |
| Any alert or rule mutation | POST, PUT, PATCH, DELETE | Yes |

The existing health response exposes alert counts and rule counts. The unauthenticated health response must be reduced to:

```json
{"status": "ok"}
```

Database status, alert counts, loaded rule counts, exception messages, file paths, and versions do not belong in an unauthenticated response.

## 2. Authorization specification

Do not add read and write scopes in this version.

One key grants access to the single operator. Adding scopes while issuing only one key with every scope is decorative authorization, not meaningful least privilege.

Endpoint-level access is still explicit:

- `/api/health` is public.
- Every data or action endpoint is protected.
- Scan and ingestion operations receive stricter rate limits and validation.

Interview explanation:

> LOTL-Detector has one operator and no multi-user trust boundary. A single opaque key is sufficient. If a remotely hosted read-only dashboard or multiple integrations were introduced, the next step would be separately generated read and write keys, not a full RBAC framework.

Do not place the API key directly in publicly hosted dashboard JavaScript. The separate static dashboard must be local, operator-controlled content. A public dashboard build cannot safely hold a shared secret.

## 3. CORS specification

### Default policy

No CORS origins are allowed unless explicitly configured.

Default:

```python
CORS_ORIGINS = []
```

Configured local dashboard origins:

```text
http://localhost:8080
http://127.0.0.1:8080
```

Origins must be exact scheme, hostname, and port tuples. Do not use:

- `*`
- Regular expressions
- Arbitrary subdomains
- Origin reflection
- `null`

### Exact policy

```python
CORS(
    app,
    resources={r"/api/*": {
        "origins": app.config["CORS_ORIGINS"],
        "methods": ["GET", "POST", "OPTIONS"],
        "allow_headers": ["Authorization", "Content-Type"],
        "expose_headers": ["Retry-After", "X-Request-ID"],
        "supports_credentials": False,
        "max_age": 3600,
    }},
)
```

Exact settings:

| Property | Value |
|---|---|
| Origins | Configured dashboard origins only |
| Methods | `GET`, `POST`, `OPTIONS` |
| Request headers | `Authorization`, `Content-Type` |
| Exposed headers | `Retry-After`, `X-Request-ID` |
| Cookie credentials | Disabled |
| Preflight cache | 3600 seconds |

CORS is not authentication. It prevents untrusted browser origins from reading responses or making permitted credentialed requests. Non-browser clients are unaffected.

Requests with an unapproved `Origin` should still be processed according to normal HTTP authentication, but must receive no `Access-Control-Allow-Origin` header. CORS enforcement is performed by the browser.

## 4. Network binding specification

Default:

```text
127.0.0.1:5000
```

Change `core/config.py`:

```python
"api": {
    "host": "127.0.0.1",
    "port": 5000,
    "debug": False,
}
```

Resolution order:

1. `LOTL_API_HOST`
2. `api.host` from configuration
3. `127.0.0.1`

Port resolution:

1. `LOTL_API_PORT`
2. `api.port`
3. `5000`

### Development server

The current line is unsafe:

```python
app.run(debug=True, host="0.0.0.0", port=5000)
```

Replace it with configuration-driven startup and never hard-code debug mode.

Refuse to start if both conditions are true:

- Debug mode is enabled
- Host is not `127.0.0.1` or `::1`

The Werkzeug debugger can provide code execution. It must never be exposed to a network.

### Remote deployment

If remote access is required:

1. Bind the application server to `127.0.0.1` or a private Unix socket.
2. Put nginx, Caddy, or another reverse proxy in front.
3. Terminate TLS at the proxy.
4. Restrict ingress with a host firewall.
5. Preserve the client IP only from a trusted proxy.
6. Run with Gunicorn or Waitress, not `app.run()`.

The bearer key is plaintext to anyone who can observe an HTTP connection. Plain HTTP is acceptable only over loopback. TLS is mandatory for network deployment.

## 5. SQLite concurrency specification

### Current defect

`AlertDatabase` currently maintains one connection:

```python
self.connection
```

and disables SQLite's thread check:

```python
check_same_thread=False
```

This does not make the connection thread-safe. It only disables the safety exception. Concurrent request threads can interleave transactions and cursor operations on the same connection.

### Required pattern

Use one SQLite connection per Flask request, stored in `flask.g`.

- Open lazily on first database use.
- Reuse for the remainder of that request.
- Close during `teardown_appcontext`.
- Keep `check_same_thread=True`, which is the default.
- Do not use a global connection.
- Do not create a connection pool. SQLite connections are cheap and a pool adds no value here.

### Connection configuration

Every connection:

```python
sqlite3.connect(
    database_path,
    timeout=5.0,
    detect_types=sqlite3.PARSE_DECLTYPES | sqlite3.PARSE_COLNAMES,
)
```

Then:

```sql
PRAGMA foreign_keys = ON;
PRAGMA busy_timeout = 5000;
```

At database initialization:

```sql
PRAGMA journal_mode = WAL;
PRAGMA synchronous = NORMAL;
```

Exact choices:

| Setting | Value |
|---|---:|
| Journal mode | WAL |
| Busy timeout | 5000 milliseconds |
| Foreign keys | ON |
| Synchronous mode | NORMAL |
| Thread bypass | Removed |
| Connection lifetime | One request |

`journal_mode=WAL` is persistent for the database file and should be established during schema initialization. `busy_timeout` and `foreign_keys` are connection-specific and must be set on every new connection.

### Transaction pattern

Read methods do not explicitly begin a transaction.

Each write operation uses a short explicit transaction:

```python
with connection:
    cursor = connection.execute(...)
```

A scan that creates multiple alerts should use one transaction for the entire alert batch, not one commit per alert:

```python
with connection:
    alert_ids = db.save_alerts(connection, alerts)
```

Do not keep a database transaction open while parsing logs or running the detection engine. Perform collection and detection first, then open the short write transaction.

### Data-access structure

Keep `AlertDatabase` as a stateless repository:

```python
class AlertDatabase:
    def get_alerts(self, connection, ...):
        ...

    def save_alerts(self, connection, alerts):
        ...
```

It must not own a Flask request connection.

Schema initialization should use a temporary standalone connection and close it immediately.

### Adjacent fix

Add:

```python
get_alert_by_id(connection, alert_id)
```

The current implementation loads up to 10,000 alerts and searches in Python. That is inefficient and creates unnecessary exposure under load.

## 6. Input validation specification

Use `jsonschema`, which is already present in `requirements.txt`, for JSON request bodies. Use explicit parsing functions for query strings.

All validation errors return:

```json
{
  "error": "validation_error",
  "fields": {
    "limit": "must be between 1 and 500"
  }
}
```

Do not include stack traces or raw schema exceptions.

### Global request rules

For JSON POST endpoints:

- Require `Content-Type: application/json`.
- Reject malformed JSON with `400`.
- Reject unsupported media types with `415`.
- Reject unknown object properties.
- Reject duplicate query parameters.
- Reject unknown query parameters.
- Do not coerce booleans, arrays, or objects into strings.
- Set `MAX_CONTENT_LENGTH` to 1,048,576 bytes.

### `GET /api/alerts`

Allowed parameters:

| Parameter | Rule |
|---|---|
| `start_time` | ISO 8601 timestamp with timezone |
| `end_time` | ISO 8601 timestamp with timezone |
| `severity` | `critical`, `high`, `medium`, `low` |
| `platform` | `windows`, `linux`, `macos` |
| `min_score` | Integer from 0 through 150 |
| `limit` | Integer from 1 through 500, default 100 |
| `offset` | Integer from 0 through 100,000, default 0 |

Additional rules:

- Reject unknown query parameters.
- Reject repeated parameters such as `limit=10&limit=20`.
- Require both timestamps to contain timezone information.
- Normalize valid timestamps to UTC.
- Require `start_time <= end_time`.
- Allow filters to be combined.
- Apply `limit` and `offset` to every filter path.

The current implementation ignores `limit` when `severity`, `platform`, or `min_score` is supplied. Replace the separate query methods with one parameterized query builder so every filter composes correctly.

Use parameterized SQL only. Values must never be interpolated into SQL strings.

### `GET /api/alerts/<alert_id>`

Flask route converter:

```python
<positive_int:alert_id>
```

Or validate after `<int:alert_id>`:

```text
alert_id >= 1
```

A syntactically valid but nonexistent ID returns `404`.

### `GET /api/rules`

Current endpoint has no inputs.

If filtering is added, allow only:

| Parameter | Rule |
|---|---|
| `platform` | `windows`, `linux`, `macos` |
| `severity` | `critical`, `high`, `medium`, `low` |

Do not add rule mutation endpoints in this session. Rule files are executable detection policy and require more careful validation, atomic replacement, and audit logging than a basic CRUD route.

If rule management is later introduced:

- Accept YAML or JSON, never arbitrary Python.
- Validate against the existing rule schema.
- Reject unknown keys.
- Limit rule body to 64 KiB.
- Require rule IDs to match `^[A-Z]{2,8}-[0-9]{3,6}$`.
- Write to a temporary file.
- Revalidate and reload all rules.
- Replace atomically with `os.replace()`.
- Preserve the previous version if reload fails.
- Disallow client-provided output paths.

### `GET /api/stats`

No parameters and no request body.

Reject any unexpected query parameter with `400`.

### `POST /api/scan`

Exact schema:

```python
SCAN_SCHEMA = {
    "type": "object",
    "required": ["platform", "log_path"],
    "additionalProperties": False,
    "properties": {
        "platform": {
            "type": "string",
            "enum": ["windows", "linux"],
        },
        "log_path": {
            "type": "string",
            "minLength": 1,
            "maxLength": 1024,
        },
    },
}
```

Path validation:

1. Require an absolute path.
2. Resolve with `Path.resolve(strict=True)`.
3. Require a regular file or directory.
4. Reject symbolic links for the submitted path.
5. Require containment under one configured `SCAN_ALLOWED_ROOTS` entry.
6. Use `Path.is_relative_to()` for containment.
7. Do not use string prefix checks.
8. Require the selected platform's supported file type.
9. Reject device files, sockets, FIFOs, and other special files.

Example:

```python
resolved = candidate.resolve(strict=True)

if candidate.is_symlink():
    reject()

if not any(resolved.is_relative_to(root) for root in allowed_roots):
    reject()

if not resolved.is_file() and not resolved.is_dir():
    reject()
```

Default allowed scan roots:

```text
tests/fixtures
~/.local/share/lotl-detector/imports
```

Resolve these roots at startup.

Do not default to `/var/log`. Reading broad system log trees can expose unrelated secrets and can make a demo scan unexpectedly expensive.

Resource constraints:

- Maximum files per directory scan: 1,000
- Maximum size per file: 100 MiB
- Maximum combined input size: 500 MiB
- Reject files that grow or change identity during validation and opening where practical
- Stop processing once a configured limit is reached

### Future `POST /api/ingest`

If added:

- JSON object or JSON array only
- Maximum body: 1 MiB
- Maximum events: 1,000
- Maximum command line: 4,096 characters
- Maximum process name: 256 characters
- Maximum username: 256 characters
- Timestamp must be timezone-aware ISO 8601
- Platform enum only
- Reject unknown fields
- Batch insert in one short transaction

### `POST /api/alerts`

The inspected server does not currently expose this route. Do not invent it merely because the initial summary suggested one.

If external alert creation is genuinely required, treat it as ingestion and validate it with the same constraints. Otherwise, alerts should be created only by the detection engine.

## 7. Rate limiting specification

Use Flask-Limiter. Do not write a security-sensitive concurrent limiter using a module-level dictionary.

Add a pinned Flask-Limiter dependency compatible with the project environment.

### Algorithm and storage

- Strategy: moving window
- Storage: in-memory
- Key: authenticated key identity when authentication succeeds, otherwise remote IP
- Testing: limiter explicitly disabled through `RATELIMIT_ENABLED=False`
- Response: JSON `429` with `Retry-After`

In-memory storage is intentionally limited to one process. This is acceptable for the documented single-process local deployment.

If the application later runs multiple workers, use Redis-backed limiter storage. Otherwise, each worker enforces a separate quota.

### Exact limits

| Endpoint | Limit |
|---|---:|
| `GET /api/health` | 60 per minute |
| `GET /api/alerts` | 120 per minute |
| `GET /api/alerts/<id>` | 120 per minute |
| `GET /api/rules` | 60 per minute |
| `GET /api/stats` | 30 per minute |
| `POST /api/scan` | 6 per minute |
| Future `POST /api/ingest` | 30 per minute |
| Future alert or rule mutation | 30 per minute |

The scan limit is only an abuse backstop. It does not provide job scheduling or guarantee that concurrent scans are safe.

Add a process-local scan semaphore with capacity 1:

- If no scan is running, start it.
- If another scan is running, return `409 Conflict`.
- Always release the semaphore in `finally`.

This prevents the local operator from accidentally launching overlapping CPU, disk, and database-heavy scans.

## 8. Request and timeout limits

### Request body

```python
app.config["MAX_CONTENT_LENGTH"] = 1_048_576
```

Return JSON for `413 Request Entity Too Large`.

### HTTP timeout

The Flask development server is not a deployment server and does not offer a robust application request timeout.

For the documented Waitress deployment:

```text
channel_timeout = 120 seconds
```

For a reverse proxy:

```text
client header timeout = 10 seconds
client body timeout = 30 seconds
upstream response timeout = 120 seconds
```

Exact application operation limits:

| Operation | Limit |
|---|---:|
| SQLite lock wait | 5 seconds |
| HTTP header receipt | 10 seconds |
| HTTP request body receipt | 30 seconds |
| Synchronous scan response | 120 seconds |
| Request body | 1 MiB |

A Python thread cannot safely terminate arbitrary collector code after a timeout. Therefore, the first implementation should enforce file and scan bounds and use the server timeout as a connection bound.

If scans commonly exceed 120 seconds, the correct next design is an asynchronous scan job endpoint, not a larger timeout:

```text
POST /api/scans
GET /api/scans/<id>
```

That is outside this one-session implementation.

## 9. Security headers

Apply to every API response, including errors:

```text
Content-Security-Policy: default-src 'none'; frame-ancestors 'none'; base-uri 'none'; form-action 'none'
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Referrer-Policy: no-referrer
Cache-Control: no-store
Pragma: no-cache
X-XSS-Protection: 0
Cross-Origin-Resource-Policy: same-site
```

Do not add HSTS for localhost HTTP. HSTS is meaningful only when the API is consistently served over HTTPS. For remote TLS deployment, add at the reverse proxy:

```text
Strict-Transport-Security: max-age=31536000; includeSubDomains
```

Do not rely on setting `Server` to an empty string inside Flask. The WSGI or reverse-proxy server can add its own header after Flask returns the response. Strip or replace `Server` at the deployment layer.

The dashboard needs a separate CSP because it serves HTML and JavaScript. The API CSP intentionally permits no executable content.

## 10. Logging specification

### Request log fields

Log one structured event after each request:

- UTC timestamp
- Random request ID
- Client IP
- HTTP method
- Route template when available
- Response status
- Duration in milliseconds
- Response size
- Authentication result: success, missing, invalid, or not required
- Rate-limit result
- Scan ID if applicable

Return the request ID:

```text
X-Request-ID: <random identifier>
```

Accept an incoming request ID only if it matches:

```text
^[A-Za-z0-9]{8,64}$
```

Otherwise generate a new one.

### Security event logs

Log at `WARNING`:

- Missing or invalid API key
- Rate limit exceeded
- Rejected scan path
- Unsupported media type
- Oversized request
- Attempted debug mode on a non-loopback host
- Rejected unknown or repeated parameters

Log at `INFO`:

- Server startup, host and port only
- Scan started
- Scan completed
- Number of events processed
- Number of alerts generated
- Alert IDs and rule IDs
- Database schema initialization

Log at `ERROR`:

- Database failures
- Collector failures
- Unhandled exceptions
- Rule loading failures

### Never log

- API key
- `Authorization` header
- Request body
- Response body
- Full command lines
- Raw event data
- Environment variables
- Cookies
- Query strings as one raw string
- Unvalidated paths
- Stack traces in HTTP responses

For validation errors, log field names and categories, not submitted values:

```text
validation_failed fields=log_path,platform
```

For scan operations, log either:

- A validated path relative to an allowed scan root, or
- A short hash of the resolved path

Do not return `str(exception)` to clients for `500` responses. The current server does this in several handlers and can expose internal paths or implementation details.

Client response:

```json
{
  "error": "internal_server_error",
  "request_id": "..."
}
```

Full exception details remain in server logs.

## 11. Concrete Python pseudocode

### `api/auth.py`

```python
import hmac
import logging
from functools import wraps

from flask import current_app, jsonify, request

logger = logging.getLogger(__name__)


def extract_bearer_token():
    values = request.headers.getlist("Authorization")

    if len(values) != 1:
        return None

    scheme, separator, token = values[0].partition(" ")

    if separator != " " or scheme.lower() != "bearer":
        return None

    if not token or any(character.isspace() for character in token):
        return None

    return token


def require_api_key(view):
    @wraps(view)
    def wrapped(*args, **kwargs):
        expected = current_app.config["API_KEY"]
        provided = extract_bearer_token()

        valid = (
            provided is not None
            and hmac.compare_digest(
                provided.encode("utf-8"),
                expected.encode("utf-8"),
            )
        )

        if not valid:
            logger.warning(
                "authentication_failed method=%s path=%s remote_addr=%s",
                request.method,
                request.path,
                request.remote_addr,
            )

            response = jsonify({"error": "unauthorized"})
            response.status_code = 401
            response.headers["WWW-Authenticate"] = "Bearer"
            return response

        return view(*args, **kwargs)

    return wrapped
```

### Secured application factory

```python
import logging
import os
import secrets
import threading
import time
from pathlib import Path

from flask import Flask, g, jsonify, request
from flask_cors import CORS
from flask_limiter import Limiter

from api.auth import require_api_key
from core.database import close_db, get_db, initialize_database

logger = logging.getLogger(__name__)


def create_app(test_config=None):
    app = Flask(__name__)

    loaded = load_project_config()

    app.config.from_mapping(
        DATABASE_PATH=loaded["database"]["path"],
        RULES_DIR=loaded["rules"]["directory"],
        LOG_LEVEL=loaded["logging"]["level"],
        API_HOST=os.getenv(
            "LOTL_API_HOST",
            loaded["api"].get("host", "127.0.0.1"),
        ),
        API_PORT=int(os.getenv(
            "LOTL_API_PORT",
            loaded["api"].get("port", 5000),
        )),
        API_DEBUG=loaded["api"].get("debug", False),
        CORS_ORIGINS=loaded["api"].get("cors_origins", []),
        SCAN_ALLOWED_ROOTS=loaded["api"].get(
            "scan_allowed_roots",
            [
                "tests/fixtures",
                "~/.local/share/lotl-detector/imports",
            ],
        ),
        MAX_CONTENT_LENGTH=1_048_576,
        RATELIMIT_ENABLED=True,
    )

    if test_config:
        app.config.update(test_config)

    app.config["API_KEY"] = load_api_key(app.config)

    if (
        app.config["API_DEBUG"]
        and app.config["API_HOST"] not in {"127.0.0.1", "::1"}
    ):
        raise RuntimeError(
            "Debug mode cannot be used on a non-loopback host"
        )

    app.extensions["rule_loader"] = build_rule_loader(
        app.config["RULES_DIR"]
    )
    app.extensions["engine"] = build_detection_engine(
        app.extensions["rule_loader"]
    )
    app.extensions["collectors"] = build_collectors()
    app.extensions["scan_lock"] = threading.BoundedSemaphore(1)

    initialize_database(app.config["DATABASE_PATH"])
    app.teardown_appcontext(close_db)

    CORS(
        app,
        resources={
            r"/api/*": {
                "origins": app.config["CORS_ORIGINS"],
                "methods": ["GET", "POST", "OPTIONS"],
                "allow_headers": [
                    "Authorization",
                    "Content-Type",
                ],
                "expose_headers": [
                    "Retry-After",
                    "X-Request-ID",
                ],
                "supports_credentials": False,
                "max_age": 3600,
            }
        },
    )

    limiter = Limiter(
        key_func=rate_limit_identity,
        strategy="moving-window",
        storage_uri="memory://",
    )
    limiter.init_app(app)
    app.extensions["limiter"] = limiter

    @app.before_request
    def begin_request():
        supplied_id = request.headers.get("X-Request-ID")

        if supplied_id and valid_request_id(supplied_id):
            g.request_id = supplied_id
        else:
            g.request_id = secrets.token_hex(8)

        g.request_started = time.monotonic()

    @app.after_request
    def secure_response(response):
        response.headers["X-Request-ID"] = g.request_id
        response.headers["Content-Security-Policy"] = (
            "default-src 'none'; "
            "frame-ancestors 'none'; "
            "base-uri 'none'; "
            "form-action 'none'"
        )
        response.headers["X-Content-Type-Options"] = "nosniff"
        response.headers["X-Frame-Options"] = "DENY"
        response.headers["Referrer-Policy"] = "no-referrer"
        response.headers["Cache-Control"] = "no-store"
        response.headers["Pragma"] = "no-cache"
        response.headers["X-XSS-Protection"] = "0"
        response.headers["Cross-Origin-Resource-Policy"] = "same-site"

        duration_ms = round(
            (time.monotonic() - g.request_started) * 1000,
            2,
        )

        logger.info(
            "request_completed request_id=%s method=%s "
            "path=%s status=%s duration_ms=%s remote_addr=%s",
            g.request_id,
            request.method,
            request.path,
            response.status_code,
            duration_ms,
            request.remote_addr,
        )

        return response

    @app.errorhandler(413)
    def request_too_large(_error):
        return jsonify({
            "error": "request_too_large",
            "request_id": g.request_id,
        }), 413

    @app.errorhandler(429)
    def rate_limited(error):
        response = jsonify({
            "error": "rate_limit_exceeded",
            "request_id": g.request_id,
        })
        response.status_code = 429
        response.headers["Retry-After"] = str(
            exact_retry_after(error)
        )
        return response

    @app.errorhandler(500)
    def internal_error(error):
        logger.exception(
            "unhandled_exception request_id=%s",
            g.request_id,
        )
        return jsonify({
            "error": "internal_server_error",
            "request_id": g.request_id,
        }), 500

    register_routes(app, limiter)

    return app
```

### Protected route examples

```python
def register_routes(app, limiter):
    @app.get("/api/health")
    @limiter.limit("60 per minute")
    def health():
        return jsonify({"status": "ok"})

    @app.get("/api/alerts")
    @require_api_key
    @limiter.limit("120 per minute")
    def alerts():
        filters = validate_alert_query(request.args)
        connection = get_db()
        records = AlertDatabase.get_alerts(
            connection,
            **filters,
        )
        return jsonify({
            "count": len(records),
            "alerts": records,
        })

    @app.post("/api/scan")
    @require_api_key
    @limiter.limit("6 per minute")
    def scan():
        require_json_content_type()
        payload = validate_scan_request(request.get_json())

        scan_lock = app.extensions["scan_lock"]

        if not scan_lock.acquire(blocking=False):
            return jsonify({
                "error": "scan_already_running",
            }), 409

        try:
            collector = app.extensions["collectors"][
                payload["platform"]
            ]
            events = collector.collect_events(
                payload["log_path"]
            )
            alerts = app.extensions["engine"].match_events(
                events
            )

            connection = get_db()

            with connection:
                alert_ids = AlertDatabase.save_alerts(
                    connection,
                    alerts,
                )

            return jsonify({
                "events_processed": len(events),
                "alerts_generated": len(alerts),
                "alert_ids": alert_ids,
            })
        finally:
            scan_lock.release()
```

Decorator order matters. Authentication should execute before the endpoint-specific rate limiter identity needs the authenticated key. Verify actual Flask-Limiter decorator behavior during implementation and test unauthorized traffic separately.

## 12. Component lifetime specification

Replace module-level globals:

```python
db
rule_loader
engine
collectors
```

with:

```python
app.extensions["rule_loader"]
app.extensions["engine"]
app.extensions["collectors"]
app.extensions["scan_lock"]
```

Database connections belong in `g`.

This prevents `create_app()` calls in tests from overwriting global components used by another application instance.

The rule loader and engine may remain application-scoped because they are initialized once and primarily read after startup. If rule mutation is added later, reload them under a lock and replace the complete engine reference atomically.

## Interview summary

The concise defense is:

> I used a high-entropy opaque bearer key because LOTL-Detector is a single-operator API with no user identity model. I protect all detection data and action endpoints, leave only a minimal health check public, bind to loopback by default, and require TLS for remote deployment. CORS is restricted to configured dashboard origins. Inputs are allowlisted and size-bounded, scans are constrained to configured roots, and expensive endpoints are rate-limited. SQLite uses one connection per request with WAL and a bounded lock wait rather than sharing one connection across threads. Logs preserve security events and request correlation while excluding credentials, bodies, and command lines. I deliberately omitted OAuth, JWTs, and RBAC because the threat model does not justify them.

This is secure enough to demonstrate competent engineering without cosplaying as an identity platform.
